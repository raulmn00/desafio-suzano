# OVGS — Plano de Implementação

> **Para workers agênticos:** este plano é executado fase a fase. Steps usam checkbox (`- [ ]`). Cada módulo de backend segue o MESMO padrão Clean Architecture do módulo de referência (`clientes`).

**Goal:** Construir o Sistema de Gestão de Ordens de Venda (OVGS) — backend NestJS (Clean Architecture + Prisma + Postgres) e frontend React/Vite — com ≥95% de cobertura, documentação e deploy (Cloud Run gen2 + Vercel).

**Architecture:** Monólito modular em Clean Architecture. Domínio puro (entidades, value objects, máquina de estados, erros, ports) → aplicação (use-cases) → infraestrutura (NestJS controllers, Prisma repos, DI). Auditoria append-only gravada na mesma transação da mudança de estado (estilo outbox).

**Tech Stack:** Node 22, TypeScript, NestJS 11, Prisma 6, PostgreSQL 16, Jest (Istanbul), Swagger, JWT/Passport, React 19 + Vite 7, TanStack Query, Zod, react-hook-form, Cypress (component + e2e), Docker Compose, GitHub Actions, Cloud Run gen2 (functions-framework) + Neon, Vercel.

---

## Princípios de execução
- **DRY / YAGNI / TDD** — domínio e use-cases sempre com teste antes ou junto da implementação.
- **Commits granulares** (Conventional Commits), **sem** `Co-authored-by`.
- Cada fase termina com testes verdes e (quando aplicável) `pnpm test:cov` ≥ thresholds.
- `coverageProvider: 'babel'` (Istanbul). Threshold: 95% global, **100% em `domain/` e `application/use-cases/`**.

## Fases

### Fase 1 — Fundação backend
- [ ] NestJS app: package.json (deps + scripts test/test:cov/build), tsconfig, nest-cli.json, eslint/prettier
- [ ] Jest config (Istanbul, threshold 95/100, collectCoverageFrom, ignore module/dto/main)
- [ ] Prisma schema (todos os modelos) + datasource Neon-ready (url + directUrl)
- [ ] Shared kernel: `Either`/`Result`, `DomainError`, base `Entity`, `UniqueEntityId`, `Clock` port + `SystemClock`, `IdGenerator` + `UuidGenerator`
- [ ] Infra compartilhada: `PrismaService`, `DomainExceptionFilter` (mapeia DomainError→HTTP), `AppModule`, `main.ts` (dev), `function.ts` (gen2)
- [ ] Health module + Swagger setup
- [ ] `docker-compose.yml` (Postgres) na raiz
- [ ] Commit por bloco coeso

### Fase 2 — Módulo de referência: `clientes`
- [ ] Domínio: `Cliente` (nome, documento, transportes autorizados), VOs (`Documento`), errors, `ClienteRepository` (port) — com specs (invariantes + autorizar/desautorizar transporte)
- [ ] Aplicação: use-cases `CriarCliente`, `EditarCliente`, `ConsultarClientes`, `ConsultarClientePorId`, `AutorizarTipoTransporte` — specs com `InMemoryClienteRepository`
- [ ] Infra: controller + HTTP DTOs (class-validator), `PrismaClienteRepository` + mapper, módulo (DI via `useClass`/`useFactory`)
- [ ] Specs de controller (use-case mockado)
- [ ] `pnpm test:cov` verde no módulo

### Fase 3 — Cadastros: `tipos-transporte` e `itens`
- [ ] `tipos-transporte`: CRUD (criar/editar/consultar), código único, extensível. Domínio+app+infra+testes (paralelizável via subagente seguindo padrão `clientes`)
- [ ] `itens`: criar/consultar, SKU único. Idem.

### Fase 4 — Núcleo: `ordens-venda` + Central de Agendamento
- [ ] Domínio: `StatusOrdemVenda` (enum) + `MaquinaEstadosOV` (tabela de transições) com specs `it.each` (válidas + inválidas)
- [ ] `OrdemDeVenda` (root): invariantes (1 cliente, 1 transporte, ≥1 item), `transicionarPara`, `agendar/reagendar/confirmarAgendamento`, `alterarTransporte`
- [ ] `Agendamento` (VO/entidade): dataEntrega, janela (início/fim), confirmado
- [ ] Use-cases: `CriarOrdemVenda` (valida transporte autorizado p/ cliente), `ConsultarOrdens` (filtros status/cliente/transporte/data), `ConsultarOrdemPorId`, `AtualizarStatus`, `DefinirAgendamento`, `ConfirmarAgendamento`, `Reagendar` — todos emitindo AuditEvent na mesma transação
- [ ] Infra: controller, DTOs, `PrismaOrdemVendaRepository` + mapper, módulo
- [ ] Specs completas (100% domínio/use-cases)

### Fase 5 — Auditoria
- [ ] Domínio: `AuditEvent` (ator, ação, alvo, estado-antes/depois, timestamp, correlationId) append-only
- [ ] `AuditLogger` port (shared/application) + impl Prisma; integração no `TransactionManager`
- [ ] Use-case `ConsultarAuditoria` (filtros entidade/ação/período) + controller (AUDITOR)
- [ ] Specs

### Fase 6 — Auth (JWT leve + RBAC)
- [ ] `AuthModule`: `POST /auth/login`, JWT strategy, `JwtAuthGuard` (global) + `@Public()`, `RolesGuard` + `@Roles()` (OPERADOR/AUDITOR)
- [ ] Seed de usuários (operador/auditor) no `prisma/seed.ts`
- [ ] Specs (guards unit + e2e login)

### Fase 7 — Integração & e2e & cobertura
- [ ] Testes de integração de repositórios (Postgres via docker compose) e e2e (supertest): fluxo completo de uma OV CRIADA→ENTREGUE + auditoria
- [ ] Fechar cobertura ≥95% global / 100% núcleo; ajustar exclusões
- [ ] `prisma/seed.ts` com dados de demonstração

### Fase 8 — Frontend (React + Vite)
- [ ] Scaffold Vite+TS, libs (axios, TanStack Query, Zod, react-hook-form, router), `lib/http.ts`, `lib/env.ts`
- [ ] Features: `auth`, `clientes`, `tipos-transporte`, `itens`, `ordens-venda` (gestão), `monitoramento` (filtros), `agendamento` (central)
- [ ] Tratamento de estados (loading/erro), validações de entrada (zod)
- [ ] Cypress: `vite-plugin-istanbul` + `@cypress/code-coverage` + `nyc`; component tests + e2e (com API mockada via `cy.intercept`)

### Fase 9 — Infra, CI/CD, deploy
- [ ] `docker-compose.yml` final (Postgres + backend) + scripts de execução
- [ ] GitHub Actions `ci.yml` (lint+typecheck+test:cov backend; build+cypress frontend)
- [ ] Deploy backend → Cloud Run function gen2 (Neon, migrate deploy) ; `deploy.yml`
- [ ] Deploy frontend → Vercel (Root Dir frontend, VITE_API_URL)

### Fase 10 — Documentação & release
- [ ] `README.md` raiz + `backend/README.md` + `frontend/README.md` (instruções, tecnologias, decisões arquiteturais, modelagem de domínio, persistência, escalabilidade, performance, trade-offs — estilo DECISÃO & JUSTIFICATIVA)
